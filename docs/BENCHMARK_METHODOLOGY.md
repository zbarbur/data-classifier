# Benchmark Methodology

> **Version:** 1.0
> **Updated:** 2026-04-11 (Sprint 4)

## Overview

The data_classifier benchmark suite measures detection quality at multiple levels, using both synthetic and real-world data. Benchmarks run manually (not in CI) to avoid slow test suites and external data dependencies.

## Metrics

### Primary Metrics

| Metric | Level | Definition | Why |
|---|---|---|---|
| **Macro F1** | Column / Entity | Average of per-entity-type F1 scores | Treats rare and common entity types equally. Prevents high-volume types from masking poor performance on rare types. |
| **Micro F1** | Column / Entity | F1 computed from aggregate TP/FP/FN across all types | Reflects overall volume-weighted accuracy. Good for "how often are we right?" |
| **Primary-Label Accuracy** | Column | Percentage of columns where the top prediction (highest confidence) matches ground truth | Measures the user-visible outcome: when a column gets one label, is it correct? |

### Secondary Metrics

| Metric | Level | Definition |
|---|---|---|
| Precision | Per-entity | TP / (TP + FP) — how many detections are correct |
| Recall | Per-entity | TP / (TP + FN) — how many real instances are found |
| F1 | Per-entity | Harmonic mean of precision and recall |
| Sample match rate | Per-entity | What fraction of sample values matched the regex pattern |
| Sample validation rate | Per-entity | What fraction of matched samples passed validation (Luhn, checksum, etc.) |

### Secret Detection Metrics

| Metric | Level | Definition |
|---|---|---|
| Per-layer P/R/F1 | Detection layer | Breaks down accuracy by detection mechanism (regex, scanner_definitive, scanner_strong, scanner_contextual) |
| Overall P/R/F1 | Aggregate | Combined across all layers |
| FP breakdown | Per-TN-category | Which categories of true negatives trigger false positives |

## Evaluation Levels

### 1. Pattern-Level (regex only)

Tests individual regex patterns against raw string values. No column context, no column-name engine, no orchestrator.

- **Input:** `(value_string, expected_entity_type)` pairs
- **Engine:** RegexEngine only, applied to each value individually
- **Output:** Per-pattern TP/FP/FN, collision matrix

### 2. Column-Level (full pipeline)

Tests the complete classification pipeline: column-name engine + regex engine + orchestrator + confidence scoring.

- **Input:** `ColumnInput` objects with multiple sample values, labeled with expected entity type
- **Engine:** Full `classify_columns()` pipeline
- **Output:** Per-entity metrics, engine contribution breakdown, sample-level analysis

### 3. Secret Detection (credential-specific)

Tests the secret/credential detection pipeline specifically, including known-prefix regex, scanner tiers, and entropy analysis.

- **Input:** Individual string values labeled as secret/not-secret, tagged with expected detection layer
- **Engine:** Full pipeline, checking for CREDENTIAL entity type
- **Output:** Per-layer P/R/F1, FP/FN breakdown

## Corpus Sources

### Synthetic (default)

Generated by `corpus_generator.py` using Faker. Advantages: deterministic, covers all entity types, includes format variations and adversarial negatives. Disadvantage: may not reflect real-world distribution.

### Real-World (optional)

Loaded by `corpus_loader.py` from external corpus samples. Sources:
- **Ai4Privacy pii-masking-300k** — 225K rows, 27+ PII types
- **Nemotron-PII** — 100K records, 55+ types
- **SecretBench** — 97K labeled secrets
- **gitleaks fixtures** — test cases from gitleaks rules
- **detect-secrets test suite** — test fixtures from Yelp's detect-secrets

See `docs/research/SECRET_CORPORA_RESEARCH.md` for full corpus documentation.

## How to Run

```bash
# Column-level accuracy (synthetic corpus, 100 samples/type)
python -m tests.benchmarks.accuracy_benchmark --samples 100

# Column-level accuracy (real-world corpus)
python -m tests.benchmarks.accuracy_benchmark --corpus ai4privacy --max-rows 500

# Secret detection benchmark
python -m tests.benchmarks.secret_benchmark

# Full sprint report (all benchmarks combined)
python -m tests.benchmarks.generate_report --sprint 4 --samples 500
```

## Interpreting Results

### Macro F1 vs Micro F1

- **Macro F1 < Micro F1**: Some rare entity types have poor detection. Investigate per-entity breakdown.
- **Macro F1 > Micro F1**: High-volume entity types drag down the average. May need pattern tuning for common types.
- **Both high (>0.9)**: Good overall detection quality.

### Primary-Label Accuracy

This is the most user-visible metric. If a connector asks "what type is this column?", primary-label accuracy measures how often we get it right. Target: >90%.

### Per-Entity F1

Look for entity types with:
- **Low recall, high precision**: Missing patterns or too-strict regex. Add patterns or relax validation.
- **High recall, low precision**: Over-matching. Tighten regex or add validation.
- **Low F1 overall**: Fundamental detection problem. May need new engine or approach.
