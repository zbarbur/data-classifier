# Migration Notes: `data_classifier` v0.8.0 → v0.15.0

> **Audience:** BigQuery-connector integration team.
> **Written:** 2026-04-24.
> **Jump size:** 7 sprints, 4 published minor versions (v0.8.0 → v0.12.0 → v0.13.0 → v0.14.0 → v0.15.0).

## TL;DR

- **No Python API breakage.** Every v0.8.0 call site still compiles and runs unchanged.
- **One behavior change worth re-goldening:** the **meta-classifier directive flip in Sprint 14** changed the primary-label decision for a minority of `structured_single` columns (accuracy improved from family-macro F1 0.83 → 0.95; some specific entity-type labels on individual columns will differ). Re-run your golden fixtures after the upgrade.
- **Drop the Dockerfile URL override.** v0.15.0's `download_models.py` default URL is correct. The `--url` / `--checksum-url` scaffolding added for the v0.8.0 wheel bug can come out in the same PR that bumps the pin.
- **New capabilities worth adopting** (additive, no pressure): `ClassificationFinding.family` for family-level policy, `scan_text()` for free-text scanning, `health_check()` for readiness probes.

---

## What forced this migration

The v0.8.0 wheel published to AR was built from a pre-fix commit and serves a broken `download_models.py` default URL (`docker.pkg.dev/data-classifier-prod/...` — a path that never existed). The release pipeline's existence-based idempotency check prevented the subsequent fixed build from replacing it. Root cause, fix, and post-publish content-verified idempotency are all in v0.15.0's release pipeline (`cloudbuild-release.yaml`).

## Integration checklist

### 1. Bump the version pin

```dockerfile
# Before (v0.8.0):
RUN pip install "data_classifier[ml]==0.8.0" ...

# After (v0.15.0):
RUN pip install "data_classifier[ml]==0.15.0" ...
```

### 2. Remove the `download_models` URL override

```dockerfile
# Drop all of this — the default URL in v0.15.0 is correct:
#   ARG MODEL_VERSION=urchade-gliner-multi-pii-v1
#   RUN python -m data_classifier.download_models \
#       --version "${MODEL_VERSION}" \
#       --url "https://artifactregistry.googleapis.com/v1/..." \
#       --checksum-url "..."

# Replace with:
RUN python -m data_classifier.download_models
```

### 3. Re-run your golden fixtures

Expect deltas in ~5–15% of column-level primary labels due to the directive flip. Expected direction: more accurate (higher F1 on the family benchmark). Capture the diff and treat it as ground truth for your downstream assertions.

### 4. Verify engines loaded correctly

Call `data_classifier.get_active_engines()` at service startup. Expected output in v0.15.0:

```python
[
    {"name": "column_name",    "order": 10, "class": "ColumnNameEngine"},
    {"name": "regex",          "order": 20, "class": "RegexEngine"},
    {"name": "heuristic",      "order": 30, "class": "HeuristicEngine"},
    {"name": "secret_scanner", "order": 40, "class": "SecretScannerEngine"},
    {"name": "gliner2",        "order": 50, "class": "GLiNER2Engine"},
]
```

If `gliner2` is missing, the `[ml]` extras aren't installed or the ONNX model wasn't baked in. If you're intentionally regex-only, set `DATA_CLASSIFIER_DISABLE_ML=1` and assert the first four engines are present.

### 5. (Optional) Adopt family-level policy

If your downstream DLP logic has hand-maintained `entity_type → policy` mappings, consider switching to `finding.family` for the coarse-grained decision. 13 families, stable, auto-populated, and covers every entity type.

```python
# Before:
if finding.entity_type in {"CREDIT_CARD", "IBAN", "SWIFT_BIC", "ROUTING", ...}:
    ...

# After:
if finding.family == "FINANCIAL":
    ...
```

## Behavior changes that could shift results

### Sprint 14 — Meta-classifier directive flip (v0.14.0, live)

The meta-classifier v5 (shadow since Sprint 12) is now the **live directive** on `structured_single` columns. Measured impact on our family-accuracy benchmark: `family_macro_f1` 0.8300 → 0.9509, `cross_family_rate` 0.1624 → 0.0915. 14 entity types at perfect F1.

For your golden fixtures, expect:
- Some columns that returned `entity_type=X` previously will now return `entity_type=Y` in the same family (sibling shift, lower cost).
- Fewer cross-family mistakes (e.g. a username column being labeled EMAIL).
- `free_text_heterogeneous` and `opaque_tokens` column shapes are **unaffected** — the meta-classifier is gated out on those branches via the Sprint 13 column-shape router.

### Sprint 12 — DOB_EU entity type retired from emission (v0.12.0)

If your consumer code pattern-matched on `entity_type == "DOB_EU"`, update to `entity_type == "DOB"`. The **family alias is preserved** (`family == "DATE"`), so family-level logic keeps working without changes.

### Sprint 14 — CREDENTIAL split into subtypes in benchmark GT (v0.14.0)

The benchmark ground truth now splits `CREDENTIAL` into `API_KEY` / `PRIVATE_KEY` / `OPAQUE_SECRET`. Runtime emission has used these subtypes since Sprint 8 (v0.8.0) — this is a benchmark-methodology change only, not an API change. Your existing consumer code that already handles the subtypes is unchanged.

### Sprint 15 — Confidence values will shift (v0.15.0)

The scoring model was rethought in Sprint 15. Three changes affect the `ClassificationFinding.confidence` value (the field shape is unchanged):

1. **Validated matches floor at `0.95`.** Findings whose pattern passes a structural validator (Luhn for credit cards, EIP-55 for Ethereum, base58check for Bitcoin, checksum for VIN/ABA/etc.) now get `confidence >= 0.95` regardless of prior computation. Previously a validated match could land anywhere on the scale.
2. **The "count multiplier" is removed.** Findings no longer get a confidence boost proportional to the number of matched sample values. One strong match and ten weak matches now produce comparable confidence.
3. **Char-class diversity boost added.** Values with high character-class evenness (mix of letters/digits/symbols) now get a small confidence lift — helps with opaque-token identification.

**Impact on BQ:**
- Consumer code with hard-coded confidence thresholds (e.g. `if finding.confidence > 0.80: block()`) should re-validate against the new distribution. Expected direction: validated matches move up, non-validated matches roughly unchanged.
- Relative sort order within a column's findings may shift — if downstream logic depends on "highest-confidence finding," the identity of that finding could change.
- The `confidence` field type, range (`[0.0, 1.0]`), and semantics (higher = more confident) are unchanged.

## New capabilities (additive, safe to adopt at your pace)

| Sprint | Feature | Import | Notes |
|---|---|---|---|
| 11 | `ClassificationFinding.family` | existing | New field on every finding. 13 families. |
| 11 | `FAMILIES`, `family_for()`, `ENTITY_TYPE_TO_FAMILY` | `from data_classifier import ...` | Taxonomy helpers for downstream policy. |

| 14 | `health_check()` probe | `from data_classifier import health_check` | Safe for `/health` endpoints — never raises. |

## What **didn't** change

Just to make the "no API breakage" claim concrete:

- `classify_columns()` signature — unchanged.
- `ColumnInput`, `ColumnStats`, `ClassificationFinding`, `SampleAnalysis` dataclass shapes — additive fields only (`family`).
- `load_profile()`, `load_profile_from_yaml()`, `load_profile_from_dict()` — unchanged.
- `compute_rollups()`, `rollup_from_rollups()` — unchanged.
- Sensitivity levels: `LOW < MEDIUM < HIGH < CRITICAL` — unchanged.
- 4 category names: `PII`, `Financial`, `Credential`, `Health` — unchanged.

## Verification

After deploying v0.15.0, run this one-column smoke probe on startup:

```python
from data_classifier import health_check

result = health_check()
assert result["healthy"], f"classifier unhealthy: {result['error']}"
assert "gliner2" in result["engines_executed"], "GLiNER2 not loaded — check ONNX bake"
assert result["latency_ms"] < 500, f"probe slow: {result['latency_ms']:.1f}ms"
```

If any assertion fails, the Docker image is misconfigured (missing extras, missing ONNX model, broken download-models bake).

## Rollback plan

The v0.8.0 wheel with the broken `download_models.py` URL will remain on AR unless explicitly yanked. If you need to roll back for any reason, the `--url` / `--checksum-url` override you added for the v0.8.0 workaround still works.

We're keeping the stale v0.8.0 wheel in place (rather than yanking it) because yanking would break any `pip install` that already resolved its hash. Once you're on v0.15.0, there should be no consumer pinned to v0.8.0, and we can yank it as a post-sprint cleanup.
